# Data Quality Release Pipeline

Automated pipeline that updates the Data Quality documentation pages when a new OMOP release ships. One `ingest` + one `generate` = all pages updated.

## What it updates

| Page | What changes |
|------|-------------|
| **Data Quality Results** | New release section with pass rate, failure counts, and diff from previous release |
| **Known Issues** | Open/Resolved issue lists regenerated from full DQD history |
| **DBT Tests** | Test definition tables regenerated from catalog + manifest; release changelog updated |
| **Data Quality landing** | Pass rate badge updated with latest release tag |

## Prerequisites

- `uv` installed ([docs](https://docs.astral.sh/uv/))
- AWS CLI configured (for S3 downloads)
- Working directory: repo root (`emory-omop/`)

## Step-by-step: New release

Replace `v1.1.0` with the actual release version throughout.

### 1. Download DQD results

DQD results are generated by the `e_omop_dqd` R project after each ETL run.

```bash
# Source directory (local)
ls ~/git_repos/lits__EmoryDataSolutions/e_omop_dqd/www/results/

# Find the latest file — naming is: emory-YYYYMMDDHHMMSS.json
# Example: emory-20260303193027.json
```

Copy or note the path to the new DQD JSON file.

### 2. Download DBT artifacts from S3

DBT artifacts are written to S3 by the Airflow pipeline after each ETL run.

```bash
# List available artifacts to find the latest prod run
aws s3 ls s3://416-omop-etl-hub/dbt-runs/manifest/ | grep prod | tail -5
aws s3 ls s3://416-omop-etl-hub/dbt-runs/results/ | grep prod | tail -5
aws s3 ls s3://416-omop-etl-hub/dbt-runs/catalog/ | grep prod | tail -5
```

S3 file naming convention: `{env}_{airflow_run_id}_{artifact}_{YYYYMMDD}_{HHMMSS}.{ext}`

There are **two** manifest/run_results pairs per pipeline run:
- **Model run** (earlier timestamp) — models, not tests
- **Test run** (later timestamp) — this is the one you want

Download the **test run** manifest, the **test run** run_results, and the catalog:

```bash
mkdir -p /tmp/dbt_artifacts

# Download test manifest (later timestamp)
aws s3 cp s3://416-omop-etl-hub/dbt-runs/manifest/prod_XXXXX_manifest_YYYYMMDD_HHMMSS.json \
  /tmp/dbt_artifacts/manifest.json

# Download test run_results (same timestamp as test manifest)
aws s3 cp s3://416-omop-etl-hub/dbt-runs/results/prod_XXXXX_run_results_YYYYMMDD_HHMMSS.json \
  /tmp/dbt_artifacts/run_results.json

# Download catalog (one per run, slightly later timestamp)
aws s3 cp s3://416-omop-etl-hub/dbt-runs/catalog/prod_XXXXX_catalog_YYYYMMDD_HHMMSS.json \
  /tmp/dbt_artifacts/catalog.json
```

**How to tell model vs test manifest apart**: the test manifest/results have a later timestamp. If unsure, open the file — test manifests contain nodes with `"resource_type": "test"`, model manifests contain `"resource_type": "model"`.

### 3. Ingest artifacts

From the repo root:

```bash
cd /path/to/emory-omop

uv run scripts/update_dqd_summary.py ingest --release v1.1.0 \
  --dqd ~/git_repos/lits__EmoryDataSolutions/e_omop_dqd/www/results/emory-YYYYMMDDHHMMSS.json \
  --dbt-manifest /tmp/dbt_artifacts/manifest.json \
  --dbt-results /tmp/dbt_artifacts/run_results.json \
  --dbt-catalog /tmp/dbt_artifacts/catalog.json
```

All flags are optional — you can ingest DQD-only, DBT-only, or both:

```bash
# DQD only
uv run scripts/update_dqd_summary.py ingest --release v1.1.0 \
  --dqd /path/to/dqd_results.json

# DBT only
uv run scripts/update_dqd_summary.py ingest --release v1.1.0 \
  --dbt-manifest /path/to/manifest.json \
  --dbt-results /path/to/run_results.json \
  --dbt-catalog /path/to/catalog.json
```

**Expected output:**

```
Ingested DQD /path/to/dqd.json -> scripts/dqd_history/v1.1.0.json
  2,374 checks, 97.2% pass rate
Ingested DBT manifest (133 test nodes) -> scripts/dbt_history/v1.1.0/tests.json
Ingested DBT run_results (133 test results) -> scripts/dbt_history/v1.1.0/results.json
Ingested DBT catalog (13 tables) -> scripts/dbt_history/v1.1.0/catalog.json
  133 tests across 13 tables — 133 pass, 0 fail
```

### 4. Preview changes (dry run)

```bash
uv run scripts/update_dqd_summary.py generate --dry-run
```

This prints all generated markdown to stdout without writing any files. Review the output to verify:
- New release section appears in DQD Results with correct diff
- Known Issues reflect any newly resolved or new failures
- DBT test tables are correct
- DBT changelog shows added/removed/changed tests

### 5. Generate pages

```bash
uv run scripts/update_dqd_summary.py generate
```

**Expected output:**

```
Updated: docs/Data in Enterprise OMOP/Data Quality/Data Quality Results/index.md
Updated: docs/Data in Enterprise OMOP/Data Quality/Known Issues/index.md
Updated pass rate: docs/Data in Enterprise OMOP/Data Quality/index.md
Updated: docs/Data in Enterprise OMOP/Data Quality/DBT Tests/index.md
```

### 6. Verify locally

```bash
uv run mkdocs serve
```

Check these pages in your browser:
- **Data Quality Results** — new release section expanded at top, previous releases collapsed
- **Known Issues** — Open Issues and Resolved Issues updated
- **DBT Tests** — test definition tables and changelog updated
- **Data Quality landing** — pass rate shows new release

### 7. Commit and push

```bash
git add scripts/dqd_history/ scripts/dbt_history/ docs/
git commit -m "Update data quality pages for v1.1.0"
git push
```

## Directory structure

```
scripts/
├── README.md                    ← this file
├── update_dqd_summary.py       ← the pipeline script
├── dqd_history/                 ← DQD JSON files, one per release
│   ├── v0.1.0.json
│   ├── v0.1.1.json
│   ├── v0.2.0.json
│   ├── v0.2.2.json
│   ├── v0.2.4.json
│   └── v1.0.0.json
└── dbt_history/                 ← DBT artifacts, one dir per release
    └── v1.0.0/
        ├── tests.json           ← extracted test nodes from manifest (~50KB)
        ├── results.json         ← run_results.json (test run only)
        └── catalog.json         ← catalog.json (column metadata)
```

## How it works

1. **`ingest`** copies artifacts into the history directories. The DBT manifest is stripped to test nodes only (keeping repo size small).
2. **`generate`** reads all history, sorts by semver, computes cross-release diffs (new failures, resolved, persistent for DQD; added, removed, status changes for DBT), and writes markdown between HTML comment markers in each page.
3. Pages use `<!-- MARKER_START -->` / `<!-- MARKER_END -->` comment pairs. Content between markers is regenerated; content outside markers is preserved.

## Troubleshooting

**"Markers not found" warning**: The target markdown page is missing the HTML comment markers. Check that the markers weren't accidentally deleted:
- `<!-- DQD_SUMMARY_START -->` / `<!-- DQD_SUMMARY_END -->` in Data Quality Results
- `<!-- KNOWN_ISSUES_START -->` / `<!-- KNOWN_ISSUES_END -->` in Known Issues
- `<!-- DBT_CURRENT_START -->` / `<!-- DBT_CURRENT_END -->` in DBT Tests
- `<!-- DBT_RELEASE_LOG_START -->` / `<!-- DBT_RELEASE_LOG_END -->` in DBT Tests
- `<!-- DQD_PASS_RATE_START -->` / `<!-- DQD_PASS_RATE_END -->` in Data Quality landing

**"No test nodes found in manifest"**: You downloaded the model manifest instead of the test manifest. The test manifest has the later timestamp.

**Re-running ingest**: Safe to re-run — it overwrites the existing release files. Then re-run `generate`.

**Adding a release retroactively**: Just ingest the old artifacts with the correct `--release` tag and re-run `generate`. The script rebuilds all diffs from the full history.
